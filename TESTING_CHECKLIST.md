# Multi-Bot Trading System - Testing Checklist

Complete this checklist before deploying to live trading on Coinbase.

---

## Phase 1: Setup & Initialization (Pre-Deployment)

### Environment & Configuration
- [ ] `.env` file created with all Coinbase API credentials
- [ ] Credentials verified (can call Coinbase API directly)
- [ ] `config/bots.yaml` created with single test bot
- [ ] Test allocation amount set to $100-$500
- [ ] Hard fence enabled (`type: "hard"`)
- [ ] Database URL configured and database running
- [ ] PostgreSQL database created and accessible

### Database & Migrations
- [ ] Database migration completed: `alembic upgrade 003`
- [ ] `bot_name` columns present in: trades, positions, trading_events, health_checks, bot_sessions
- [ ] No migration errors in logs
- [ ] Database tables verified: `\dt` in psql shows all expected tables

### Code & Dependencies
- [ ] Python dependencies installed: `pip install -r requirements.txt`
- [ ] Orchestrator imports correctly: `python -c "from trading-automata.orchestration.orchestrator import BotOrchestrator"`
- [ ] All Sigma Series strategies importable
- [ ] No import errors in terminal

---

## Phase 2: Local Test Run (30 Minutes)

### Startup
- [ ] Bot starts without errors: `python -m trading-automata.main`
- [ ] Output shows "Multi-bot mode detected - using BotOrchestrator"
- [ ] No exceptions in first 10 seconds
- [ ] Telegram notification received (if configured): "[bot_name] Connected to Coinbase broker"

### Broker Connection
- [ ] Broker successfully connects within 30 seconds
- [ ] Account data retrieved (cash balance, positions)
- [ ] No "Cannot connect" or "Invalid credentials" errors
- [ ] Logs show: "[bot_name] Connected to Coinbase broker successfully"

### Strategy Initialization
- [ ] All enabled strategies registered (check logs)
- [ ] Strategy configuration loaded from `config/strategies.yaml`
- [ ] No strategy initialization errors
- [ ] Each strategy shows 0 signals generated at startup (correct, need more bars)

### Data Collection
- [ ] Bars are being received (check every 1-2 minutes): `[bot_name] X bars received for SYMBOL`
- [ ] At least 3-5 bar receipts in first 5 minutes
- [ ] No "Failed to fetch bars" errors
- [ ] Timestamps progress forward (not stuck)

### Indicator Calculation
- [ ] No division-by-zero errors
- [ ] No NaN/Inf values in log output
- [ ] Event logger shows filter checks being performed
- [ ] Example: `Filter check: min_volume - PASS` or `FAIL`

---

## Phase 3: Signal Generation & Safety (24 Hours)

### First Signals
- [ ] At least one signal generated by 24 hours in (or verify expected based on strategy/market)
- [ ] Telegram notification sent for signals: "[bot_name] Signal: BUY BTC-USD @ $42,500"
- [ ] Signal includes confidence, SL, TP in metadata
- [ ] No crash on signal generation

### Order Execution (if signals trigger)
- [ ] Order successfully submitted to Coinbase
- [ ] Order ID logged in database
- [ ] Order status tracked correctly (OPEN → FILLED or CANCELLED)
- [ ] Fills recorded in `trades` table with correct `bot_name`
- [ ] Database query confirms: `SELECT * FROM trades WHERE bot_name = 'coinbase_bot'`

### Hard Fence Verification
- [ ] Create a test scenario: manually increase allocation in config above account balance
- [ ] Expected: Order rejected with log message "Hard fence: order cost $X exceeds virtual balance $Y"
- [ ] Fence does NOT allow over-allocation
- [ ] Soft fence (if enabled) allows overage up to `overage_pct` with warning logged

### Virtual Balance Tracking
- [ ] Virtual balance calculated correctly: `allocated_capital - virtual_spent + virtual_proceeds`
- [ ] After buy: `virtual_spent` increases
- [ ] After sell: `virtual_proceeds` increases
- [ ] Database query: `SELECT * FROM positions WHERE bot_name = 'coinbase_bot'` shows correct quantities

### Health Checks
- [ ] Health check recorded every 5 minutes (or configured interval)
- [ ] Status shows HEALTHY or DEGRADED (not ERROR unless exception)
- [ ] Example: `[coinbase_bot] Health check: BTC-USD | SigmaSeriesFastStrategy - HEALTHY`

---

## Phase 4: Event Logging & Debugging (24+ Hours)

### Event Log Completeness
- [ ] Query event log: `SELECT event_type, COUNT(*) FROM trading_events WHERE bot_name = 'coinbase_bot' GROUP BY event_type`
- [ ] Expected types present: bar_received, filter_checked, signal_generated, order_submitted, order_filled
- [ ] At least 100+ event_log entries after 24 hours
- [ ] No excessive ERROR entries (< 5% of total)

### Filter Statistics
- [ ] Filter pass rate: total_bars → filtered_by_min_volume → filtered_by_volatility → signal_candidates
- [ ] Expected: 30-70% pass rate (too low = filters too strict, too high = no risk control)
- [ ] Example:
  ```
  Total bars received: 1440 (1 per minute × 24 hours)
  Pass min_volume:     1200 (83%)
  Pass volatility:      800 (56%)
  Signal candidates:     45 (3% - normal)
  ```

### Error Tracking
- [ ] No repeated errors (each error logged once or infrequently)
- [ ] Errors are recoverable (connection retry, invalid bar rejected, etc.)
- [ ] No fatal errors causing bot shutdown
- [ ] Example acceptable errors:
  - "Broker disconnected, retrying with backoff"
  - "Invalid bar (close=0), skipping"
  - "Strategy not enabled, skipping"

---

## Phase 5: Performance & Monitoring (3-7 Days)

### Uptime Tracking
- [ ] Bot session recorded in database: `SELECT * FROM bot_sessions WHERE bot_name = 'coinbase_bot'`
- [ ] Session shows continuous uptime (no unexpected restarts)
- [ ] `/status` Telegram command shows uptime accurately
- [ ] Example: "Uptime: 5 days, 3 hours"

### Trade Statistics (if trades executed)
- [ ] Winning trades: at least 50%+ (break-even acceptable for early test)
- [ ] Stop loss hits: verify SL prices from metadata match actual fills
- [ ] Take profit hits: verify TP prices from metadata match actual fills
- [ ] No orphaned trades (every buy has corresponding sell or open position)

### P&L Calculation
- [ ] Run `/metrics coinbase_bot` command (if Telegram enabled)
- [ ] Chart shows:
  - Daily P&L bar chart
  - Cumulative P&L line chart
  - Win rate percentage
  - Metrics: profit_factor, sharpe_ratio, max_drawdown
- [ ] P&L matches manual calculation from trades table
- [ ] Example: 5 trades, 3 winners (+$150), 2 losers (-$100) = +$50 net

### Database Query Verification
```sql
-- Total trades by bot
SELECT bot_name, COUNT(*) FROM trades GROUP BY bot_name;

-- Open positions
SELECT * FROM positions WHERE bot_name = 'coinbase_bot' AND status = 'open';

-- Trade P&L
SELECT symbol, action, quantity, price,
  CASE WHEN action='buy' THEN -quantity*price ELSE quantity*price END as pnl
FROM trades
WHERE bot_name = 'coinbase_bot'
ORDER BY created_at;

-- Recent events
SELECT event_type, details, created_at
FROM trading_events
WHERE bot_name = 'coinbase_bot'
ORDER BY created_at DESC
LIMIT 20;
```

## Phase 6: Strategy-Specific Validation

### SigmaSeriesFastStrategy (if enabled)
- [ ] VWAP calculation converges after 5+ bars
- [ ] Volume spike detection works (>2x avg volume triggers signal)
- [ ] EMA crossover (8/21) detected correctly
- [ ] RSI(7) stays within 0-100 range
- [ ] ATR(7) scales appropriately (not 0, not infinite)
- [ ] Signals generated within 2 bars of conditions met (cooldown working)
- [ ] Expected signal frequency: 2-5 per day on liquid pairs

### SigmaSeriesAlphaStrategy (if enabled)
- [ ] Bollinger Bands upper/lower within reasonable range
- [ ] Stochastic %K crossover detected (not stuck at 0 or 100)
- [ ] EMA(50/200) slow drift (not rapid oscillation)
- [ ] RSI(14) < 30 (oversold) triggers on relevant bars
- [ ] Signals generated within 5 bars of conditions met
- [ ] Expected signal frequency: 1-3 per day

### SigmaSeriesAlphaBullStrategy (if enabled)
- [ ] EMA(21/50/200) stack verifiable in logs
- [ ] ADX(14) > 25 filters strong trends only
- [ ] RSI(10) stays in 35-55 zone before signal (not spiky)
- [ ] MACD histogram direction tracked correctly
- [ ] Exit on RSI > 72 or EMA crossdown executes
- [ ] Expected behavior: Long-only, holds through trends, exits on reversal signals
- [ ] Expected signal frequency: 0-2 per day (trend-dependent)

## Phase 7: Stress Tests & Edge Cases (Optional, Before Heavy Trading)

### Rapid Market Movement
- [ ] Test during high-volatility hours (market open, economic news)
- [ ] Verify bot doesn't generate duplicate orders
- [ ] Check for race conditions in bar processing
- [ ] Confirm hard fence still enforces during fast markets

### Connectivity Recovery
- [ ] Manually disconnect network for 30 seconds
- [ ] Verify bot attempts reconnect (logs show backoff attempts)
- [ ] After reconnect: no missed orders, consistent state
- [ ] Expected: Connection re-established within 2-3 backoff cycles

### Order Rejection Scenarios
- [ ] Manually reject first order (broker-side simulation)
- [ ] Expected: Log shows rejection, next signal can be sent
- [ ] No hung orders or orphaned positions
- [ ] Database state remains clean

### Database Failure Recovery
- [ ] Stop database connection (don't kill database, just network)
- [ ] Expected: Bot logs warnings, continues collecting bars (buffered)
- [ ] After reconnect: buffered events written correctly
- [ ] No duplicate or lost trades

---

## Phase 8: Decision Point (After 7-30 Days)

### Metrics to Evaluate
- [ ] Win rate >= 50% (break-even acceptable)
- [ ] Profit factor >= 1.0 (revenue / losses >= 1.0)
- [ ] Sharpe ratio >= 0.5 (risk-adjusted return)
- [ ] Max drawdown acceptable (< allocation amount is good)
- [ ] Strategy generating expected signal frequency for asset/timeframe

### Action Decision
- [ ] **CONTINUE**: All metrics positive, increase allocation by 50% next week
- [ ] **ADJUST**: 1-2 metrics weak, tweak risk parameters or filters
- [ ] **PAUSE**: Win rate < 40%, disable strategy and analyze
- [ ] **PIVOT**: Fundamentally misaligned, consider different strategy

### Data Retention
- [ ] Export trades, events, health checks to CSV for analysis
- [ ] Save logs for future reference
- [ ] Record decision rationale (what worked, what didn't)

## Quick Reference: Critical Error Messages

| Error | Cause | Action |
|-------|-------|--------|
| `Cannot connect to Coinbase` | Invalid API credentials or network | Verify `.env`, test API directly |
| `Hard fence: order cost $X exceeds virtual balance $Y` | Position too large for allocation | NORMAL - fence working correctly |
| `Invalid bar (close=0)` | Data feed glitch | Acceptable, bot skips automatically |
| `Broker disconnected` | Network or API timeout | Expected, bot retries automatically |
| `Strategy not enabled` | Strategy not in `config/strategies.yaml` | Verify YAML syntax and strategy name |
| `NaN/Inf in indicator` | Division by zero or data issue | Check if sufficient bars collected |
| `Order rejected by broker` | Insufficient funds or invalid params | Check Coinbase API response, retry |
