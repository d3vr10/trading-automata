# Event Logging System - Troubleshooting Guide

## Overview

The trading bot now has comprehensive **event logging** to track all trading decisions, filter checks, signals, and order events. This helps troubleshoot why the bot isn't making trades.

## How It Works

### 1. **Real-Time Process Logs** (stdout)
Every decision point is logged to console:
```
[EURUSD] Bar received: O=1.0850, H=1.0855, L=1.0845, C=1.0852
[EURUSD] ✓ PASS Volume: volume=500k >= min=500k
[EURUSD] ✓ PASS Volatility: ATR=0.0045 (min=0.003, max=0.015)
[EURUSD] 🎯 SIGNAL: BUY qty=0.5, confidence=0.95
[EURUSD] ✓ Order submitted: 12345, BUY 0.5
[EURUSD] ✓ Order FILLED: 12345, BUY 0.5 @ 1.0850
```

### 2. **Database Event Log** (trading_events table)
Important events are stored in the database for:
- Historical analysis
- Querying past decisions
- Dashboard/monitoring

## CLI Commands

### View Recent Events
```bash
# Show last 50 events
docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events

# Filter by symbol
docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events --symbol EURUSD

# Filter by type
docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events --type SIGNAL_GENERATED

# Filter by severity
docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events --severity ERROR

# Limit results
docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events --limit 100
```

## Troubleshooting Flow

**Follow this flow if the bot isn't making trades:**

```
1. Check process logs
   → docker-compose logs trading-bot -f

2. Look for "Bar received"
   ✓ YES → Bars are coming in, move to step 3
   ✗ NO → Data feed not working, check Alpaca connection

3. Look for "FILTER_PASSED" or "FILTER_FAILED"
   ✓ PASSED → Filters are OK, move to step 4
   ✗ FAILED → Filters too strict, adjust in config/strategies.yaml

4. Look for "SIGNAL_GENERATED"
   ✓ YES → Strategy is generating signals, move to step 5
   ✗ NO → Strategy conditions not met, check indicators

5. Look for "ORDER_SUBMITTED"
   ✓ YES → Order submitted to broker, move to step 6
   ✗ NO → Order manager error, check logs

6. Look for "ORDER_FILLED"
   ✓ YES → Trade completed! ✅
   ✗ NO → Order not filled, check market conditions
```

## Event Types

| Event | Meaning | Log Level |
|-------|---------|-----------|
| BAR_RECEIVED | New bar/candle processed | DEBUG |
| FILTER_PASSED | Filter check passed (volume, volatility) | DEBUG |
| FILTER_FAILED | Filter rejected the bar | WARNING |
| SIGNAL_GENERATED | Trading signal created | INFO |
| ORDER_SUBMITTED | Order sent to broker | INFO |
| ORDER_FILLED | Order executed | INFO |
| ORDER_FAILED | Order rejected by broker | ERROR |
| ERROR | Any error | ERROR |
| WARNING | Non-critical issue | WARNING |

## Example: Debugging No Trades

**Scenario:** Bot running but no trades after 24 hours

1. Check logs:
   ```bash
   docker-compose -f docker/docker-compose.yml logs trading-bot -f | grep -i "bar\|signal\|order"
   ```

2. Check filter rejections:
   ```bash
   docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events \
     --type FILTER_FAILED \
     --symbol EURUSD \
     --limit 10
   ```

   If you see: `volume=200k < min=500k`
   → Adjust `min_volume` in `config/strategies.yaml`

3. Check signal generation:
   ```bash
   docker-compose -f docker/docker-compose.yml exec trading-bot python -m src.cli events \
     --type SIGNAL_GENERATED \
     --limit 10
   ```

   If no signals:
   → Strategy conditions (RSI, EMA) not being triggered
   → Adjust strategy parameters in `config/strategies.yaml`

## Database Schema

Events stored in `trading_events` table:
- `event_type`: Type of event (BAR_RECEIVED, FILTER_PASSED, etc.)
- `event_timestamp`: When it happened
- `severity`: DEBUG, INFO, WARNING, ERROR, CRITICAL
- `symbol`: Trading symbol (e.g., EURUSD)
- `strategy`: Strategy name (e.g., rsi_atr_trend_eurusd)
- `broker`: Broker name (alpaca or coinbase)
- `message`: Human-readable message
- `details`: JSON with additional context

## Performance

- ✅ Minimal overhead (~1ms per event)
- ✅ Events logged asynchronously to database
- ✅ No impact on trading loop performance
- ✅ Graceful degradation if database unavailable
